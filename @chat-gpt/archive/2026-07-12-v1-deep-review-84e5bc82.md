# Arkini v1 deep review — follow-up

Reviewed snapshot: `84e5bc82`
Original review baseline: `6bb12839`
Review result commit: `4d116b27`

## Verdict

The implementation thread closed the important findings from the original review correctly. The fixes are not cosmetic and the architecture remains coherent. Start, resource validation, runtime mutation boundaries, identity policy, semantic authoring guards, and atomic job start all moved in the intended direction without introducing disproportionate mental load.

The original review is not fully complete. Two findings remain open: certainty-aware limited-deposit validation and structured source parse diagnostics. Job start also made the previously deferred ownership/return policies urgent before completion, cancellation, or withdrawal commands are added.

## Verified complete

### Exact start composition

`planStartFx` now returns the exact sequentially built runtime, and `startFx` commits that same runtime inside `modifyRuntimeFx`. The former proof/commit mismatch is gone.

Tests cover repeated stackable entries, stack overflow behavior, empty-runtime enforcement, and booting the current Arkini authoring config.

### Offline start bootability

`validateStartStateFx` reuses the production sequential start builder against an empty runtime with the completed config provided through `GameConfigFx`. This keeps offline and runtime semantics aligned instead of maintaining a second hand-written approximation.

The rule currently collapses a failure into one `start:invalid` diagnostic with a failure tag. That is acceptable for correctness; richer paths/messages can wait until authoring UX becomes painful.

### Resource integrity

`compileGameDirectoryFx` now reads source-aware resource descriptors and appends `validateGameResourcesFx` diagnostics to the canonical compilation result. Pack asserts the same compilation before reading bytes.

The implementation validates duplicate IDs, missing references, and unused PNGs. Asset references are exact IDs rather than inferred naming conventions.

The current config has an explicit shared blueprint tuple such as `["blueprint", "producer-lumberjack-t1"]`, matching the project decision. The missing prospector T2 resource was added.

### Swap integrity

`swapItemsFx` rejects identical runtime IDs before mutation with `SwapSameItemError`. The phantom second revision path is closed.

### Runtime architecture guards

`RuntimeBoundary.test.ts` now independently enforces:

- every public `/write/` file uses `modifyRuntimeFx`;
- revision exemptions apply only to the revision rule;
- no public write imports another public write;
- state-derived plans/results are not accepted by write commands;
- direct runtime store access and direct synchronized modification remain isolated.

### Semantic authoring guards

Owner-local line IDs and unmatched material tag selectors are now validated. The single shared `IdSchema` policy is also protected by an architecture test, preventing pseudo-domain ID wrappers from multiplying like damp basement fungus.

### Atomic line job start

The new job model is internally consistent for the implemented start phase:

- line resolution happens inside one serialized runtime transaction;
- queue capacity is checked against the locked snapshot;
- consume/reserve input operations and job creation commit atomically;
- reservations reference a persistent job ID and preserve input return location;
- concurrent starts respect queue capacity and can consume disjoint allocations;
- runtime/state round-trip includes jobs;
- runtime validation checks duplicate jobs, owner/line existence, owner grid scope, queue size, time ordering, and reservation linkage.

The new orchestration remains readable. `startLineFx` is linear and delegates allocation/application rather than becoming a mutation swamp.

## Still open from original review

### P2 — Limited deposits still use presence-only recreation

`validateLimitedDepositsFx` still puts every item ID found in outputs into one `produced` set. It does not distinguish guaranteed output from `chance: 0`, probabilistic drops, weighted alternatives, or conditional impossibility.

Therefore the original confirmed false-negative class remains. A deposit mentioned only by a never-occurring or stochastic path can suppress `deposit:unsustainable`.

Recommended next step remains intentionally narrow:

- guaranteed recreation;
- stochastic/weighted-only recreation;
- no recreation.

Do not turn this into a general progression theorem prover.

### P2 — Source parse failures are still raw defects

`readGameSourceFileFx` still executes `JSON.parse` and `GameSourceSchema.parse` inside `Effect.sync`. Malformed JSON or an invalid fragment therefore terminates source reading as a raw defect rather than producing source-aware diagnostics.

`readGameSourceFilesFx` also reads sources through a failing `Effect.forEach`, so authoring can remain one-error-per-run.

This is an authoring DX gap rather than runtime corruption, but it remains directly unfinished from the original handoff.

## New lifecycle boundary created by job start

These are not regressions in the implemented start command. They are mandatory decisions before adding completion, cancellation, or withdrawal.

### Nested ownership

An item with buffered children can still potentially become material for another owner, creating an acyclic nested ownership DAG. Starting a job now makes eventual removal semantics concrete: consuming or cancelling the parent must not orphan descendants.

The simpler recommended policy is to reject loaded owners as material unless nested machine/container behavior is deliberately wanted.

### Reserved material return

A partial reserve leaves the source remainder in the original input slot while the reserved split records that same slot as `returnLocation`. Cancellation cannot blindly restore it there.

Return behavior needs one canonical policy, preferably:

1. merge into a compatible remainder at the original input slot;
2. otherwise deterministic fallback placement;
3. otherwise typed capacity failure or retained reservation.

This policy belongs in one shared helper and must be reused by cancel/withdraw flows.

### Completion/cancel transaction integrity

Future commands must resolve live jobs inside `modifyRuntimeFx`, use job revisions, mutate reservations, remove the job, and place outputs/returns in the same transaction. No public-write composition and no UI-created completion plan.

## Mental-load assessment

The snapshot remains easier to reason about than the previous baseline. The added domains are real domains, not folder cosplay:

- resources own resource identity;
- validation rules remain one-policy files;
- jobs own persistent runs and reservations;
- writes stay thin transactional orchestrators;
- schema contracts expose state shape before behavior is implemented.

The main complexity increase is legitimate lifecycle complexity from jobs. It is still locally traceable. The dangerous point will be completion/cancellation, where placement, reservation return, job removal, output rolls, and owner validity can easily become five subtly different mutation routes. The follow-up task list explicitly guards that boundary.
